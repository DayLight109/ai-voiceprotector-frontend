# 声纹捕手 · 前端部署步骤

方案:**Next.js 标准 Node 服务**(`next start`)。前端跑一个 Node 进程(端口 3000),nginx 反代。
部署在 Web VIP `172.16.30.10`(与后端网关同机)。

## 0. 产物

| 文件 | 说明 |
|---|---|
| `sentinel-frontend.tar.gz` | 运行产物(`.next` 构建结果 + public + package.json + lock + .env.production),**不含 node_modules** |
| `sentinel-frontend.service` | systemd 单元 |
| `nginx-sentinel.conf` | nginx 反代参考(前端 + /api 后端) |

> API 地址已在构建时写死:`.env.production` 里 `NEXT_PUBLIC_API_URL=http://172.16.30.10:8080`。
> 若要改后端地址,**必须重新 build**(`NEXT_PUBLIC_` 变量编译进产物,改运行环境无效)。

## 1. 前置:服务器装 Node

`next start` 需要 Node 运行时(建议 Node 20 LTS,与构建端大版本一致)。

```bash
node -v        # 期望 v20.x 或更高；没有则先装
# Ubuntu/Debian 示例:
#   curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
#   sudo apt install -y nodejs
```

## 2. 解包 + 安装生产依赖

```bash
sudo mkdir -p /opt/sentinel-frontend
sudo tar -xzf sentinel-frontend.tar.gz -C /opt/sentinel-frontend

cd /opt/sentinel-frontend
# 只装生产依赖,用 lock 锁版本(不联网装 dev 包)
sudo npm ci --omit=dev
#   若无法用 npm ci(lock 不匹配),退而用: sudo npm install --omit=dev

# 归属给运行账户(沿用后端建的 sentinel 用户)
sudo chown -R sentinel:sentinel /opt/sentinel-frontend
```

> `npm ci --omit=dev` 会按 `package-lock.json` 精确安装。Tailwind/TS 等是 devDependencies,生产 `next start` 不需要,故 `--omit=dev`。

## 3. 装 systemd 服务

```bash
sudo cp sentinel-frontend.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now sentinel-frontend
sudo systemctl status sentinel-frontend       # 期望 active (running)
sudo journalctl -u sentinel-frontend -f        # 看到 "Ready in xxx ms" / "Local: http://localhost:3000"
```

> 单元里 `ExecStart=/usr/bin/npm start`;若你的 node/npm 不在 `/usr/bin`,用 `which npm` 查到真实路径改单元。

## 4. 验证前端进程

```bash
curl -sI http://127.0.0.1:3000 | head -1     # 期望 HTTP/1.1 200 OK
```

## 5. 配 nginx(对外入口)

把 `nginx-sentinel.conf` 内容并入你的 nginx,或放到 `/etc/nginx/conf.d/sentinel.conf`:

```bash
sudo cp nginx-sentinel.conf /etc/nginx/conf.d/sentinel.conf
sudo nginx -t            # 语法检查
sudo systemctl reload nginx
```

它做两件事:
- `/` → `127.0.0.1:3000`(前端)
- `/api/` → `127.0.0.1:8080`(后端网关,SSE 已关 buffer)

这样浏览器只访问 80/443 一个入口,前后端同源,**还能顺便消除 CORS 问题**(见下)。

## 6. 更新发布

前端改了代码、或要改后端地址,都需要**重新构建**(NEXT_PUBLIC_ 变量编译进产物):

```bash
# 本机
cd sentinel-frontend
npm run build
tar -czf deploy-dist/sentinel-frontend.tar.gz \
    .next public package.json package-lock.json next.config.ts .env.production

# 传到服务器后
sudo systemctl stop sentinel-frontend
sudo tar -xzf sentinel-frontend.tar.gz -C /opt/sentinel-frontend
cd /opt/sentinel-frontend && sudo npm ci --omit=dev
sudo chown -R sentinel:sentinel /opt/sentinel-frontend
sudo systemctl start sentinel-frontend
```

依赖没变时可跳过 `npm ci`,只覆盖 `.next` 后 `restart`。

## 7. 常见排错

| 现象 | 处理 |
|---|---|
| `npm start` 报找不到 next | 依赖没装或漏装。重跑 `sudo npm ci --omit=dev` |
| 页面能开但接口全失败 | F12 看请求地址与报错。CORS → 改后端 `.env`;连接拒绝 → 后端没起或地址错 |
| 改了后端地址不生效 | NEXT_PUBLIC_ 是编译期变量,必须重新 `npm run build` 再传 |
| 502 Bad Gateway | 前端进程没起(`systemctl status sentinel-frontend`)或端口不是 3000 |
| 静态资源 `/_next/` 404 | nginx 要反代到 `:3000`,不能当纯静态目录指 |

---

## 8. 关于 CORS(当前方案)

本次打包是**浏览器直连后端**(`.env.production` 里 `NEXT_PUBLIC_API_URL=http://172.16.30.10:8080`)。
部署后若 F12 console 报 CORS,回后端把 `.env` 的 `CORS_ALLOWED_ORIGINS` 改成浏览器地址栏的实际 origin
(如 `http://172.16.30.10`),重启网关。注意 http/https、端口必须完全一致。

> 第 5 节的 nginx 同源反代是更省心的替代方案,但当前前端代码 `API_BASE` 写死带 8080 默认值,
> 同源需改代码支持相对路径,故本版先用直连。
